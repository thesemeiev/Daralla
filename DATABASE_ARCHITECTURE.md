# 🗄️ Архитектура баз данных

## Текущее состояние

### ✅ Новые базы данных (для подписок):

1. **`subscribers.db`** - Основная БД для подписок
   - Таблица `subscribers` - пользователи Telegram
   - Таблица `subscriptions` - подписки пользователей
   - Таблица `subscription_servers` - связь подписки с серверами

2. **`notifications.db`** - Уведомления
   - Таблица `sent_notifications` - отправленные уведомления
   - Таблица `notification_effectiveness` - эффективность уведомлений
   - **Используется:** Система уведомлений для подписок

### ⚠️ Старые базы данных (из архитектуры с ключами):

3. **`vpn_keys.db`** - Платежи
   - Таблица `payments` - все платежи (pending, succeeded, canceled)
   - **Используется:** Обработка платежей (webhook'и, создание платежей)
   - **Проблема:** Название указывает на старую архитектуру с ключами
   - **Статус:** Все еще активно используется, но можно переименовать в `payments.db`

4. **`users.db`** - Пользователи и конфигурация
   - Таблица `users` - список всех пользователей (first_seen, last_seen)
   - Таблица `config` - настройки бота
   - **Используется:** 
     - `register_simple_user()` - регистрация пользователя при /start
     - `get_all_user_ids()` - получение списка пользователей для рассылки
     - `get_config()`, `set_config()` - настройки бота
   - **Проблема:** Дублирование с таблицей `subscribers` в `subscribers.db`
   - **Статус:** Все еще используется, но есть дублирование данных

## Проблемы текущей архитектуры

### 1. Дублирование пользователей

**Проблема:**
- `users.db` → таблица `users` (user_id, first_seen, last_seen)
- `subscribers.db` → таблица `subscribers` (user_id, created_at, last_seen)

**Где используется:**
- `users.db.users` → `register_simple_user()`, `get_all_user_ids()` (для рассылки)
- `subscribers.db.subscribers` → `get_or_create_subscriber()` (для подписок)

**Решение:**
- Можно объединить в одну таблицу `subscribers` в `subscribers.db`
- Или оставить как есть (разные цели: `users` для рассылки, `subscribers` для подписок)

---

### 2. Название `vpn_keys.db`

**Проблема:**
- Название указывает на старую архитектуру с ключами
- Но используется для платежей (не только для ключей)

**Решение:**
- Можно переименовать в `payments.db` (но это требует изменений в коде)
