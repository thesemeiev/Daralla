# Обработка недоступных серверов при отмене подписки

## Текущее поведение

### Что происходит, если сервер недоступен:

1. **Попытка удаления клиента:**
   - Функция `deleteClient` имеет декоратор `@retry` (3 попытки с задержкой 2 секунды)
   - Если сервер недоступен, все попытки могут провалиться
   - Исключение перехватывается в `try-except`

2. **Обработка ошибки:**
   - Сервер добавляется в список `failed_servers`
   - Ошибка логируется
   - **НО:** Связь из БД все равно удаляется

3. **Результат:**
   - ✅ Статус подписки меняется на `'canceled'`
   - ✅ Связь удаляется из БД
   - ❌ Клиент остается на недоступном сервере
   - ⚠️ Админ видит предупреждение в сообщении

## Проблемы

### 1. Несоответствие БД и серверов
- В БД связи нет
- На сервере клиент остался
- При следующей синхронизации может возникнуть конфликт

### 2. Клиент продолжает работать
- Если сервер восстановится, клиент будет работать
- Подписка отменена, но VPN работает

### 3. Нет механизма повторной попытки
- Если сервер временно недоступен, нет способа повторить удаление позже

## Текущий код обработки

```python
# 1. Удаляем клиентов со всех серверов
if server_manager and servers:
    for server_info in servers:
        server_name = server_info['server_name']
        client_email = server_info['client_email']
        
        try:
            xui, _ = server_manager.get_server_by_name(server_name)
            if xui:
                xui.deleteClient(client_email)  # Может упасть, если сервер недоступен
                deleted_servers.append(server_name)
            else:
                failed_servers.append(server_name)  # Сервер не найден
        except Exception as e:
            failed_servers.append(server_name)  # Ошибка при удалении
            logger.error(f"Ошибка удаления клиента {client_email} с сервера {server_name}: {e}")

# 2. Удаляем связи из БД (ВСЕГДА, даже если сервер недоступен)
for server_info in servers:
    await remove_subscription_server(subscription_id, server_info['server_name'])
```

## Рекомендации по улучшению

### Вариант 1: Не удалять связи из БД, если сервер недоступен (РЕКОМЕНДУЕТСЯ)

```python
# Удаляем связи только для успешно обработанных серверов
for server_info in servers:
    if server_info['server_name'] in deleted_servers:
        await remove_subscription_server(subscription_id, server_info['server_name'])
    else:
        # Оставляем связь в БД для повторной попытки
        logger.warning(f"Связь с сервером {server_info['server_name']} сохранена для повторной попытки")
```

**Плюсы:**
- Можно повторить отмену позже
- БД остается консистентной (связь есть, если клиент есть)

**Минусы:**
- Связи остаются в БД для недоступных серверов
- Нужен механизм очистки

### Вариант 2: Удалять связи, но помечать подписку как "частично отменена"

```python
# Добавить новый статус 'partially_canceled'
if failed_servers:
    await update_subscription_status(subscription_id, 'partially_canceled')
else:
    await update_subscription_status(subscription_id, 'canceled')
```

**Плюсы:**
- Видно, что отмена не завершена
- Можно добавить функцию "Завершить отмену"

**Минусы:**
- Нужно добавить новый статус
- Усложняет логику

### Вариант 3: Оставить как есть, но улучшить уведомления

**Текущее поведение приемлемо, если:**
- Админ видит список недоступных серверов
- Есть возможность повторить отмену позже
- При синхронизации система автоматически удалит "лишних" клиентов
