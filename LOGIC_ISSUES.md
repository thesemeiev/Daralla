# Проблемы в логике системы

## 🔴 Критические проблемы

### 1. Множественные активные подписки
**Проблема:** При создании новой подписки не проверяется наличие существующей активной подписки
**Где:** `bot/services/subscription_manager.py:create_subscription_for_user()`
**Последствия:**
- У пользователя может быть несколько активных подписок одновременно
- Непонятно, какая подписка используется
- При синхронизации серверов могут создаваться дубликаты клиентов

**Решение:**
```python
# Перед созданием новой подписки:
existing_sub = await get_active_subscription_by_user(user_id)
if existing_sub:
    # Вариант 1: Отменить старую подписку
    await update_subscription_status(existing_sub['id'], 'canceled')
    # Вариант 2: Продлить существующую вместо создания новой
    # await extend_existing_subscription(existing_sub['id'], days)
```

### 2. Несинхронизированное время истечения при продлении
**Проблема:** При продлении подписки:
- Время обновляется в БД (`update_subscription_expiry`)
- Ключи продлеваются на серверах (`extendClient`)
- Но время на серверах может не совпадать с БД (если продление частично не удалось)

**Где:** `bot/handlers/webhooks/payment_processors.py:process_extension_payment()`
**Последствия:**
- Расхождение между БД и реальным временем на серверах
- При синхронизации серверов может использоваться неправильное время

**Решение:**
- После продления проверять реальное время истечения на серверах
- Синхронизировать БД с максимальным временем с серверов
- Или использовать БД как источник истины и синхронизировать серверы

### 3. Создание новой подписки при существующей активной
**Проблема:** При покупке новой подписки не учитывается существующая активная подписка
**Где:** `bot/handlers/webhooks/payment_processors.py:process_new_purchase_payment()`
**Последствия:**
- Пользователь может купить новую подписку, когда старая еще активна
- Старая подписка остается активной, новая создается отдельно
- Два токена подписки для одного пользователя

**Решение:**
- Проверять наличие активной подписки перед созданием новой
- Если есть активная - либо продлевать, либо отменять старую

## 🟡 Важные проблемы

### 4. Неправильный расчет времени при синхронизации серверов
**Проблема:** В `sync_servers_with_config()`:
- Время истечения берется из БД (`expires_at_from_db`)
- Затем проверяется максимальное время на существующих серверах
- Но если подписка была продлена, а синхронизация не запускалась, время может быть устаревшим

**Где:** `bot/services/subscription_manager.py:sync_servers_with_config()`
**Последствия:**
- Новые серверы могут получить неправильное время истечения
- Клиенты на новых серверах могут истечь раньше, чем на старых

**Решение:**
- Всегда проверять реальное время на серверах перед созданием нового клиента
- Использовать максимальное из: БД, существующих серверов

### 5. Отсутствие проверки на дубликаты email при создании подписки
**Проблема:** При создании новой подписки используется `unique_email`, но не проверяется:
- Существует ли клиент с таким email на других серверах (из других подписок)
- Не конфликтует ли email с существующими клиентами

**Где:** `bot/handlers/webhooks/payment_processors.py:process_new_purchase_payment()`
**Последствия:**
- Возможны конфликты email между разными подписками
- При синхронизации может быть попытка создать клиента с существующим email

**Решение:**
- Проверять уникальность email перед созданием
- Или использовать формат `{user_id}_{subscription_id}_{uuid}` для гарантии уникальности

### 6. Частичное продление подписки
**Проблема:** При продлении подписки:
- Если продление на одном сервере не удалось, подписка все равно считается продленной
- Время в БД обновляется, но на некоторых серверах время не продлено

**Где:** `bot/handlers/webhooks/payment_processors.py:process_extension_payment()`
**Последствия:**
- Несогласованность состояния между серверами
- Пользователь может потерять доступ на части серверов

**Решение:**
- Если продление не удалось на всех серверах - откатывать обновление БД
- Или использовать компенсирующие транзакции (Saga Pattern)

### 7. Race condition при создании подписки
**Проблема:** Если два платежа обрабатываются одновременно:
- Оба могут создать подписку
- Оба могут использовать один и тот же `unique_email`
- Могут быть созданы дубликаты

**Где:** `bot/handlers/webhooks/payment_processors.py:process_new_purchase_payment()`
**Последствия:**
- Дубликаты подписок
- Конфликты при создании клиентов

**Решение:**
- Использовать транзакции с блокировками
- Проверять наличие активной подписки в транзакции
- Использовать уникальные индексы в БД (уже есть для `subscription_token`)

## 🟢 Улучшения логики

### 8. Неправильная обработка истекших подписок
**Проблема:** `get_active_subscription_by_user()` возвращает подписку со статусом 'active', но не проверяет, не истекла ли она
**Где:** `bot/db/subscribers_db.py:get_active_subscription_by_user()`
**Последствия:**
- Истекшие подписки могут считаться активными
- Пользователь может видеть истекшую подписку как активную

**Решение:**
```python
# Добавить проверку времени истечения:
WHERE sub.user_id = ? 
  AND s.status = 'active' 
  AND s.expires_at > ?
```

### 9. Несинхронизированное время при создании клиента на новом сервере
**Проблема:** В `sync_servers_with_config()` время истечения вычисляется из БД и существующих серверов, но:
- Если подписка была продлена вручную на сервере, это не учитывается
- Если время в БД устарело, новое время будет неправильным

**Решение:**
- Всегда проверять реальное время на всех существующих серверах
- Использовать максимальное время как источник истины

### 10. Отсутствие проверки консистентности данных
**Проблема:** Нет периодической проверки:
- Соответствия БД и реального состояния на серверах
- Наличия клиентов на серверах для активных подписок
- Отсутствия клиентов на серверах для неактивных подписок

**Решение:**
- Добавить периодическую задачу проверки консистентности
- Автоматически исправлять несоответствия
