1. НАЗНАЧЕНИЕ И ОБЩАЯ КАРТИНА
--------------------------------------------------------------------------------

Проект — монолитное Python-приложение для продажи и обслуживания VPN-
подписок: Telegram-бот (python-telegram-bot, long polling), HTTP-сервер
(Quart + Hypercorn) для вебхуков платёжных систем, REST API для веб-клиента
и Mini App, статика SPA из каталога apps/frontend/webapp/. Состояние хранится в SQLite
(daralla.db); панели 3x-ui обслуживаются собственным HTTP-клиентом
(apps/backend/src/daralla_backend/services/xui_panel_client.py).

Доменная модель в целом последовательная: пользователи (users, привязка
Telegram), подписки (subscriptions + subscription_servers), платежи
(payments с meta JSON), группы и узлы серверов (server_groups, servers_config).
«Источник истины» по срокам подписки — БД; фоновая синхронизация и ручной
админ-sync подтягивают панели X-UI к этому состоянию.


2. КАК ЗАПУСКАЕТСЯ И КАК СВЯЗАНЫ КОМПОНЕНТЫ
--------------------------------------------------------------------------------

Точка входа: python -m daralla_backend → apps/backend/src/daralla_backend/__main__.py → bot.bot.run().

Порядок инициализации в run():
  • Создаётся event loop, вызывается ensure_db_and_servers_ready(): миграции
    SQLite, опционально модуль событий, init_server_managers() — загрузка
    серверов из БД в MultiServerManager.
  • Собирается Telegram Application с post_init=on_startup.
  • Создаётся Quart-приложение (create_quart_app) и Hypercorn запускается в
    отдельном потоке (daemon) на 0.0.0.0:WEBHOOK_PORT.
  • Затем запускается app.run_polling() — здесь выполняется on_startup:
    NotificationManager, фоновые задачи (sync каждые 6 ч, уведомления,
    очистка платежей, снимки нагрузки), мониторинг здоровья серверов.

Важно: HTTP-слой стартует до первого тика polling и до on_startup. Часть
глобального состояния (например ctx.notification_manager) появляется только
после on_startup; критичные для оплаты сервисы (server_manager,
subscription_manager) уже заполнены после ensure_db_and_servers_ready.
Узкое окно возможно для редких гонок, если вебхук придёт в первые миллисекунды
жизни процесса до полной готовности — на практике маловероятно, но порядок
старта стоит иметь в виду при отладке.

Зависимости между сервисами инкапсулированы в AppContext (get_ctx / set_ctx) —
это лучше, чем разброс по sys.modules: явная точка доступа к менеджерам.


3. ЛОГИКА ПО КОНТУРАМ
--------------------------------------------------------------------------------

3.1. Аутентификация веб / Mini App
  • Веб: регистрация/логин (/api/auth/*), пароль через werkzeug password hash,
    сессия — случайный token в БД + опционально HttpOnly cookie
    (daralla_web_token), SameSite=Lax, secure при HTTPS.
  • API пользователя: authenticate_request_async — Bearer, cookie или initData
    Telegram; initData проверяется HMAC по документации Telegram (есть unit-
    тесты). Срок auth_date ограничен 24 часами.
  • Админ-маршруты: декоратор admin_route → require_admin: та же аутентификация
    + check_admin_access_async (ADMIN_ID из env или telegram_id пользователя).

3.2. Платежи
  • YooKassa: POST /webhook/yookassa, разбор JSON, фоновая задача
    process_payment_webhook. Идемпотентность частично обеспечена проверкой
    status/activated в БД.
  • CryptoCloud: JWT проверяется, если задан CRYPTOCLOUD_WEBHOOK_SECRET; при
    отсутствии секрета в лог пишется предупреждение, обработка статусов всё
    равно возможна — см. раздел рисков.

3.3. Подписки и X-UI
  • SubscriptionManager создаёт записи в БД, привязывает серверы группы,
    ensure_client_on_server синхронизирует клиента на панели (с локами на
    пару server_name + email).
  • SyncManager.run_sync: статусы в БД, удаление «старых» истёкших, синхронизация
    БД→панели, очистка сирот на панелях.
  • Публичная выдача конфигов: GET /sub/<token> — по токену подписки, без
    пользовательской сессии; это модель «ссылка = секрет».

3.4. Клиент apps/frontend/webapp/
  • Один крупный app.js (тысячи строк): UI, вызовы API, разбор initData из hash
    при блокировке telegram.org — продуманный fallback для Mini App.


4. СИЛЬНЫЕ СТОРОНЫ
--------------------------------------------------------------------------------

  • Чёткое разделение: handlers (Telegram), web/routes (HTTP), db/*, services/*.
  • Миграции через apps/backend/src/daralla_backend/db/migrations вместо ручного «разъехавшегося» init.
  • Тесты pytest (unit + integration): initData, парсинг YooKassa webhook,
    часть Quart и БД — база для регрессий.
  • Dockerfile с непривилегированным пользователем app.
  • Документация README с диаграммой и таблицей env.
  • Обработка продления подписки: сначала обновление expires_at в БД, затем
    синхронизация на серверы — правильный порядок для согласованности.
  • Логирование с ротацией, снижение уровня шума httpx.


5. РИСКИ (БЕЗОПАСНОСТЬ, НАДЁЖНОСТЬ, ОПЕРАЦИИ)
--------------------------------------------------------------------------------

5.1. Безопасность
  [Высокий] CryptoCloud: если CRYPTOCLOUD_WEBHOOK_SECRET не задан, постбэк
    не аутентифицируется — возможна подделка «успешной» оплаты при известном
    или угаданном идентификаторе инвойса. В продакшене секрет обязателен;
    код должен жёстко отклонять запросы без успешной верификации JWT.

  [Высокий] YooKassa: в payment.py не видно проверки подписи тела запроса /
    IP whitelist / Basic Auth, рекомендуемых или описанных в документации
    YooKassa для вебхуков. Любой, кто может отправить POST на URL, теоретически
    может дергать обработчик; защита опирается на неугадываемость payment_id
    из YooKassa. Это слабее, чем криптографическая проверка происхождения
    уведомления.

  [Средний] Логирование полного тела и заголовков YooKassa webhook (payment.py)
    — риск утечки ПДн и чувствительных метаданных в лог-файлы и централизованные
    сборщики логов. Рекомендуется логировать только id события и обезличенные
    поля.

  [Средний] CORS: Access-Control-Allow-Origin: * на auth и admin маршрутах.
    Для cookie-based сценариев браузер ограничивает кросс-сайт, но комбинация
    с Bearer в заголовке из стороннего JS (если клиент когда-либо так сделает)
    ослабляет модель. Узкий список Origin надёжнее для публичного API.

  [Средний] /sub/<token>: токен в URL — это bearer-secret. Утечка в Referer,
    прокси-логах, истории браузера даёт доступ к VPN. Это типично для subscription
    URL, но риск нужно осознавать в поддержке и коммуникациях с пользователем.

  [Низкий / операционный] GitHub Actions deploy: шаг печатает SET/NOT SET для
    имён секретов — не значения, но раскрывает, какие секреты настроены.

5.2. Данные и масштаб
  [Средний] SQLite без единого пула и без явного PRAGMA journal_mode=WAL в коде
    обзора: при росте одновременных HTTP + фоновых задач возможны блокировки
    database is locked. Для текущего масштаба часто терпимо; при росте —
    WAL, пул соединений или перенос на PostgreSQL.

  [Средний] Каждая операция в многих модулях открывает свой aiosqlite.connect —
    много коротких соединений, нет единой политики таймаутов busy_timeout.

5.3. Архитектура рантайма
  [Средний] Два asyncio event loop: основной (polling PTB + create_task из
    on_startup) и второй внутри потока Hypercorn (asyncio.run(serve(...))).
    Объекты asyncio (Lock и т.д.) нельзя смешивать между циклами; текущий код
    в основном держит HTTP-обработку и create_task вебхуков на Quart-цикле —
    это согласовано, но любая будущая «прямая» интеграция из PTB в Quart без
    очереди опасна.

  [Низкий] asyncio.create_task для вебхуков без явного «shield» от отмены при
    остановке сервера — при штатном shutdown возможны обрывы; для платежей
    лучше иметь ретраи на стороне платёжки или идемпотентную догрузку.


6. НЕДОСТАТКИ И ТЕХНИЧЕСКИЙ ДОЛГ
--------------------------------------------------------------------------------

  • Огромный apps/frontend/webapp/app.js — сложно сопровождать, ревьюить, тестировать;
    отсутствие модульной сборки/TypeScript увеличивает риск регрессий.

  • Дублирование CORS_HEADERS в нескольких модулях (api_auth vs admin_common).

  • Зависимости в requirements.txt без жёстких верхних границ для quart,
    hypercorn, pytest — воспроизводимость сборок со временем плавает.

  • flake8 в CI: жёсткая проверка только на E9,F63,F7,F82; сложность и длина
    строки с --exit-zero не ломают сборку — мёртвый код и сложность накапливаются.

  • Глобальный флаг _bootstrap_completed в startup.py защищает от двойного
    ensure_db, но логика «кто первый вызвал» размазана между run() и on_startup.

  • Часть «старых» глобальных имён в bot.py (notification_manager = None)
    дублирует идею AppContext — потенциальная путаница для новых разработчиков.


7. ПОТЕНЦИАЛЬНЫЕ ОШИБКИ И КРАЕВЫЕ СЛУЧАИ
--------------------------------------------------------------------------------

  • Продление подписки (process_extension_payment): сначала в БД вызывается
    update_subscription_expiry, затем цикл синхронизации с панелями. Если ни
    один сервер не синхронизировался, платёж помечается failed, но отката
    expires_at в БД нет — получается «оплата в платёжке прошла / в БД срок уже
    увеличен / в payments — failed / панели старые». Это баг с точки зрения
    бухгалтерии и поддержки; либо транзакция БД «срок + платёж» атомарно,
    либо при полном провале откатывать expires_at и эскалировать вручную.

  • CryptoCloud: ветка «payment not found» всё равно отвечает 200 — корректно
    для шумных постбэков, но усложняет диагностику на стороне провайдера.

  • Тестовое покрытие: хорошие точечные тесты, но нет явного e2e на полный
    сценарий «оплата → запись в БД → вызов X-UI» (моки панели частично могут
    быть в unit test_x3_sync и др. — при расширении стоит проверить).

  • YooKassa Configuration.account_id/secret_key выставляются при импорте
    bot.bot — при unit-тестах без env возможны побочные эффекты или пустые
    ключи (сейчас компенсируется тестовым окружением).


8. РЕКОМЕНДАЦИИ ПО ПРИОРИТЕТАМ
--------------------------------------------------------------------------------

  1) Включить обязательную верификацию вебхуков YooKassa и CryptoCloud (секрет,
     подпись или официальный SDK-приёмник) и отказаться от обработки при
     провале проверки.

  2) Убрать или сильно сократить логирование сырых webhook-тел и заголовков.

  3) Для SQLite: PRAGMA journal_mode=WAL; busy_timeout; при росте нагрузки —
     план миграции на клиент-серверную СУБД.

  4) Сузить CORS до доверенных origin продакшена.

  5) Декомпозиция webapp (модули, сборщик, минимум статический анализ JS).

  6) Ужесточить CI: хотя бы предупреждения flake8 без exit-zero для новых
     файлов или подключить ruff/mypy для apps/backend/src/daralla_backend/.


9. ЗАКЛЮЧЕНИЕ
--------------------------------------------------------------------------------

Проект представляет собой зрелую по структуре коммерческую платформу с
понятным потоком «оплата → БД → синхронизация X-UI», разумным разделением
слоёв и заделом на тесты. Основные вопросы к продакшену — аутентичность
вебхуков платёжных систем, утечки в логах, масштабирование SQLite и
сопровождение очень большого фронтенд-файла. Устранение пунктов раздела 8
значительно снизит операционные и финансовые риски.
