# Где мы сейчас и что делать дальше

Кратко: что уже приведено в порядок, что можно не трогать, и один понятный следующий шаг.

---

## Что уже сделано (можно не трогать)

| Часть | Что сделано |
|-------|-------------|
| **Конфиг** | Всё из env в одном месте: `bot/config.py`. Пути, токены, Remnawave — только там. |
| **Контекст** | Один объект `AppContext` в `bot/context.py`. Сервисы и настройки берутся из него (или из Flask). |
| **Подписки** | Только Remnawave. Вся логика в `subscription_service.py` + `remnawave_service.py`. Оплата вызывает одну функцию: `activate_subscription_after_payment()`. |
| **Оплата** | `payment_processors.py` только обновляет платёж в БД, вызывает subscription_service и шлёт уведомление. Без деталей Remnawave. |
| **Админка** | Маршруты разнесены по папке `admin/`: auth, users, subscription, stats, broadcast, servers. |
| **Remnawave-режим** | Если Remnawave включён — SubscriptionManager и SyncManager не создаются (None). Старый код с X-UI не дергается. |

То есть: **фундамент, подписки, оплата и админка уже разделены и работают в одну сторону (Remnawave).**

---

## Что выглядит «смешанным» (и почему так можно оставить)

- **`get_bot_module()`** — остался для обратной совместимости. Часть кода берёт сервисы из контекста, часть — из модуля `bot.bot`. Это не ломает логику, просто два способа доступа.
- **`get_globals()`** — в трёх местах (start_handler, payment_processors, error_handler) с разным набором полей. Нужен только там, где реально используется (например, error_handler — только ADMIN_IDS). Остальное можно постепенно убрать, когда будете трогать эти экраны.
- **Глобальные переменные в `bot.py`** (`server_manager`, `subscription_manager`, …) — дублируют то, что лежит в `app_context`. Нужны, пока кто-то читает их через `getattr(bot_module, ...)`. Не мешают, если не трогать.

Итого: **«смесь» — это в основном обратная совместимость. Поведение бота уже опирается на новую структуру (config → context → subscription_service → Remnawave).**

## Что делать прямо сейчас (один понятный шаг)

**Вариант А — просто пользоваться ботом**  
Ничего не менять. Запускать, тестировать оплату и админку. Всё «старое/новое» под капотом уже согласовано с Remnawave.

**Вариант Б — один следующий шаг для порядка**  
Свести «кто откуда берёт сервисы» к одному способу:

1. Открыть `bot/context.py` и `docs/ARCHITECTURE.md`.
2. Решить: «Всё, что нужно handlers и webhook, пусть берётся только из `get_app_context()`».
3. По одному файлу за раз: где есть `get_bot_module()` или свой `get_globals()` — заменить на `get_app_context()` и поля контекста. Следующий раз можно продолжить с того же списка.

**Вариант В — зафиксировать текущее состояние**  
Сделать теги в git: например, `v2-remnawave-refactor`. Дальше любые правки делать уже от этой версии, не думая про «старое/новое».

## Куда смотреть, если нужно что-то конкретное

- **Изменить настройки бота** → `bot/config.py` и `.env`
- **Логика подписок / продление после оплаты** → `bot/services/subscription_service.py`
- **Работа с Remnawave API** → `bot/services/remnawave_service.py`
- **Обработка успешной оплаты** → `bot/handlers/webhooks/payment_processors.py` (вызов `activate_subscription_after_payment` и уведомления)
- **Админ-маршруты** → `bot/handlers/webhooks/routes/admin/` (по файлам: users, servers, stats и т.д.)
- **Старт бота и фоновые задачи** → `bot/bot.py` и `bot/core/startup.py`, `bot/core/tasks.py`

## Итог

Не нужно «размазывать» рефакторинг по всему проекту. Основное уже разделено: конфиг, контекст, Remnawave, подписки, оплата, админка. То, что кажется смешанным — по большей части старые точки входа для совместимости; их можно чистить по мере касания файлов.
